# AI Spec 协作模板

## 文档职责

这份模板用于在开始一个新需求、新功能、新页面或新内容模块前，先把人类判断、待澄清问题、AI 追问、技术方案和验收标准沉淀到文档中。

它适合用于 Codex、Claude Code、ChatGPT 等 AI 协作场景。目标不是把需求一次写完，而是建立一个「人类提供判断，AI 负责追问、整理、扩展和落地」的协作流程。

使用方式：

1. 复制本模板为新的 spec 文档。
2. 先填写已知内容，空白处可以留空。
3. 让 AI 补充「AI问题列表」和「AI反问与挑战」。
4. 人类回答关键问题。
5. AI 根据回答补充技术方案、最终决策记录和实现清单。

# 需求

这次要做什么？

希望解决什么问题？

这次需求不包含什么？

# 业务背景

为什么现在要做？

现状是什么？

如果不做，会有什么问题？

这个需求和当前产品、业务、内容或工程目标有什么关系？

# 我已确定的判断

这里写已经确定、不希望 AI 随意改动的判断。

例如：

- 继续使用静态部署。
- 不引入数据库。
- 内容要适合官网展示。
- 页面需要兼顾后续扩展。

# 我还不确定的地方

这里写还没有想清楚、希望 AI 帮忙追问或推演的地方。

例如：

- 这个页面的第一目标用户是谁？
- 首篇文章应该偏技术复盘，还是偏公众号表达？
- 页面要不要展示代码仓库？
- 这次是否需要一步做到详情页？

# 目标澄清

这个需求最想达成的结果是什么？

什么结果出现时，我会认为这一版是成功的？

如果只能优先做好一件事，那应该是什么？

# 用户与场景

这个需求主要给谁使用或观看？

用户进入这个页面或使用这个功能时，最关心什么？

用户完成什么动作后，这个需求才算有价值？

# 内容边界

哪些内容必须写？

哪些内容可以延后？

哪些内容不能公开？

哪些内容容易写成自嗨，需要 AI 帮忙审稿？

# 技术约束

当前必须遵守哪些技术前提？

例如：

- 静态部署。
- 不使用数据库。
- 不新增复杂依赖。
- 保持现有设计系统。
- 保持浏览器兼容约定。

# 方案取舍

如果有多个方案，分别是什么？

每个方案的优点是什么？

每个方案的风险是什么？

当前更倾向哪个方案，为什么？

# AI反问与挑战

AI 可以在这里指出：

- 需求里哪里不清楚。
- 哪些假设可能不成立。
- 哪些内容可能不适合对外表达。
- 哪些方案可能过度设计。
- 哪些地方需要先做小版本验证。

# AI补充的技术方案

AI 在这里补充可执行技术方案。

建议包含：

- 数据结构。
- 文件结构。
- 页面结构。
- 组件拆分。
- 依赖选择。
- 静态生成或运行时逻辑。
- SEO 和 sitemap 处理。
- 浏览器兼容注意事项。
- 风险和替代方案。

# 最终决策记录

这里沉淀最终已经确认的判断。

建议写清：

- 选择了什么方案。
- 为什么选择这个方案。
- 明确不选择什么方案。
- 这个决策后续什么时候需要重新评估。

# 实现清单

- [ ] 补充内容资料。
- [ ] 确认技术方案。
- [ ] 实现数据结构。
- [ ] 实现页面或功能。
- [ ] 补充样式和响应式适配。
- [ ] 补充 SEO。
- [ ] 本地验证。
- [ ] 构建验证。
- [ ] 更新相关文档。

# 验收标准

- 功能或页面可以正常访问。
- 内容符合需求目标。
- 关键用户路径可用。
- 静态构建通过。
- 类型检查通过。
- 没有引入需求外的复杂依赖。
- 浏览器兼容敏感样式已检查。
