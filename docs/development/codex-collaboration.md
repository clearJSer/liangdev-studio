# Codex 协作开发指南

本文档的职责：帮助项目维护者用更稳定、可复盘、低返工的方式与 Codex 协作开发 `liangdev-studio`，并把日常开发流程标准化为可执行范式。

## 1. 核心原则

### 小步闭环

一次只让 Codex 处理一个清晰目标，例如：

- 实现一个页面区块
- 修复一个明确 bug
- 抽取一个共享组件
- 补齐某个模块的类型或测试

避免一次性提出“把官网做完”“全面优化项目”这类范围过大的请求。范围越大，Codex 越难判断优先级，也越容易产生不必要改动。

### 先对齐，再修改

在修改现有文件前，先让 Codex 说明：

- 它准备改哪些文件
- 为什么要改这些文件
- 是否存在多种实现方案
- 每种方案的风险和取舍

如果只是新增独立文档、示例或草稿，可以直接生成；如果会影响已有代码路径，应先对齐方案。

### 以验收标准驱动开发

每个任务都应该有明确的完成标准。推荐格式：

```text
目标：
在 web 首页新增一个案例展示区块。

约束：
- 使用 TypeScript
- 不引入新 UI 框架
- 优先复用 packages/ui
- 保持移动端可读

验收：
- pnpm lint 通过
- pnpm check-types 通过
- 首页在 375px 和 1440px 下布局正常
```

### 保持改动可审查

优秀的 AI 协作不是让 Codex 一次性生成大量代码，而是让它交付小而完整、容易 review 的变更。

推荐一次任务控制在：

- 1 个功能点
- 1 到 5 个相关文件
- 1 套明确验收命令

如果超过这个范围，先让 Codex 拆任务。

## 2. 本项目上下文

本项目是 Turborepo monorepo，使用 pnpm workspace。

```text
apps/
  web/                  Next.js 应用，默认端口 3000
docs/
  development/          开发流程、AI 协作、工程实践文档
packages/
  ui/                   共享 React 组件库，直接导出源码
  eslint-config/         共享 ESLint 配置
  typescript-config/     共享 TypeScript 配置
specs/
  web/                  官网相关需求、页面规划、内容规格
```

关键约束：

- 包管理器使用 `pnpm@9.0.0`
- Node.js 版本要求 `>=18`
- 代码使用 TypeScript
- 不允许使用 `any`
- 每个函数必须有 JSDoc 注释
- 新增文件必须在文件顶部或文档开头说明职责
- 修改现有文件前，应先说明修改意图

常用命令：

```bash
pnpm install
pnpm dev
pnpm build
pnpm check-types
pnpm lint
pnpm format
turbo dev --filter=web
turbo build --filter=web
turbo lint --filter=@repo/ui
```

文档目录约定：

- `AGENTS.md`：AI agent 自动读取的仓库规则，作为唯一事实来源。
- `CLAUDE.md`：Claude Code 入口文件，只通过 `@AGENTS.md` 继承规则。
- `docs/`：长期维护的工程文档、开发指南、架构说明。
- `docs/development/`：开发流程、协作范式、工程实践。
- `specs/`：产品需求、页面规划、业务规则、设计规格。
- `specs/web/`：官网相关需求、页面信息架构、内容草案。
- `apps/web/`：运行时源码、应用局部配置和只服务于该 app 的实现文件。

## 3. 推荐协作流程

### 第一步：描述业务目标

不要只描述技术动作，要说明业务意图。

较弱的请求：

```text
帮我改一下首页样式。
```

更好的请求：

```text
帮我把首页改成适合个人开发工作室的官网首屏。目标用户是想找独立开发者做网站的小企业主。风格要专业、清爽，不要营销感太重。
```

### 第二步：让 Codex 先读代码

推荐提示词：

```text
先阅读相关文件，不要立刻修改。请告诉我当前实现结构、你认为要改哪些文件，以及是否有多个方案。
```

适合用于：

- 页面结构调整
- 组件抽取
- 配置改造
- 影响多个包的改动

### 第三步：确认方案

当 Codex 给出方案后，你可以选择：

```text
采用方案 A，只改 apps/web，不动 packages/ui。
```

或：

```text
我希望这个组件未来可复用，请采用共享组件方案，放到 packages/ui。
```

### 第四步：执行实现

推荐提示词：

```text
按刚才确认的方案实现。修改前先说明即将改动的文件和原因；实现后运行 pnpm lint 和 pnpm check-types。
```

### 第五步：验收与复盘

完成后要求 Codex 输出：

- 改了什么
- 为什么这样改
- 跑了哪些命令
- 是否有未完成事项
- 下一步建议

推荐提示词：

```text
请总结本次改动，列出涉及文件、验证结果、潜在风险和下一步建议。
```

## 4. 任务拆分范式

### 页面开发

推荐拆分：

1. 明确页面目标和受众
2. 定义信息架构
3. 实现静态结构
4. 加入响应式样式
5. 抽取可复用组件
6. 运行 lint、类型检查和视觉检查

提示词模板：

```text
我要开发 [页面名称]。
目标用户是 [用户类型]。
页面目标是 [转化/展示/操作目标]。
请先给出信息架构和组件拆分，不要修改代码。
```

### 组件开发

推荐拆分：

1. 明确组件职责
2. 确定 props API
3. 判断是否放入 `packages/ui`
4. 实现组件
5. 在消费方接入
6. 检查类型和可访问性

提示词模板：

```text
我要新增一个 [组件名] 组件，用于 [使用场景]。
请先判断它应该放在 apps/web 内部还是 packages/ui，并说明理由。
```

### Bug 修复

推荐拆分：

1. 描述复现步骤
2. 提供实际结果和预期结果
3. 让 Codex 定位原因
4. 确认修复方案
5. 实现最小修复
6. 添加或说明验证方式

提示词模板：

```text
这里有一个 bug：
复现步骤：
1. ...
2. ...

实际结果：
...

预期结果：
...

请先定位原因，不要直接修改代码。
```

### 重构

重构要格外强调边界。不要让 Codex 做泛泛的“优化代码”。

推荐提示词：

```text
请只重构 [具体模块/文件]，目标是 [减少重复/提升类型安全/拆分组件]。
不要改变用户可见行为。
修改前请说明方案和风险。
```

## 5. 高质量提示词模板

### 新功能模板

```text
我要实现 [功能名称]。

业务目标：
[为什么要做]

范围：
- 包含：[包含内容]
- 不包含：[明确排除内容]

技术约束：
- 使用 TypeScript
- 不使用 any
- 每个函数写 JSDoc
- 优先复用现有结构

验收标准：
- [命令或行为]
- [页面/组件表现]

请先阅读相关代码，给出方案，不要立即修改。
```

### 修改现有代码模板

```text
请修改 [文件/模块]，目标是 [具体目标]。

修改前请先告诉我：
- 你打算改哪些文件
- 为什么要改
- 是否有多种方案
- 推荐方案是什么
```

### 代码审查模板

```text
请 review 当前改动，优先关注：
- bug
- 类型问题
- 运行时风险
- 架构边界
- 缺失的测试或验证

请按严重程度排序，并给出文件和行号。
```

### 提交前检查模板

```text
请做一次提交前检查：
- 查看 git diff
- 运行 pnpm lint
- 运行 pnpm check-types
- 总结是否可以提交
```

## 6. Codex 应该优先遵守的工程习惯

### 文件定位

优先使用：

```bash
rg
rg --files
```

不要让 Codex 全仓库盲目阅读。好的做法是先搜索入口，再阅读相关文件。

### 依赖选择

新增依赖前必须先说明：

- 为什么现有能力不够
- 新依赖解决什么问题
- 是否会影响 bundle 体积
- 是否有维护风险

前端视觉或交互不应轻易引入大型库。

### 类型设计

类型应该表达业务含义，而不是简单迎合编译器。

推荐：

```ts
type ServiceCard = {
  title: string;
  description: string;
  href: string;
};
```

避免：

```ts
type Data = Record<string, string>;
```

### 组件边界

放在 `packages/ui` 的组件应该是跨应用可复用的基础组件。

只服务于 `apps/web` 某个页面的组件，应优先放在 `apps/web` 内部，避免共享包变成杂物间。

### 样式策略

页面级布局优先留在页面或应用内部；基础按钮、卡片、文本展示等可复用 UI 才进入 `packages/ui`。

样式变更必须检查：

- 移动端宽度
- 桌面宽度
- 文本是否溢出
- 交互状态是否清晰
- 颜色是否过于单一

## 7. 你应该避免的协作方式

### 范围过大

不推荐：

```text
帮我把项目整体优化一下。
```

推荐：

```text
请只检查 apps/web 首页，找出结构、样式和类型上的前三个问题，先不要修改。
```

### 验收不清

不推荐：

```text
做好看一点。
```

推荐：

```text
视觉目标是简洁、可信、适合个人开发工作室。请避免大面积渐变和营销式大卡片，移动端首屏需要直接看到服务内容。
```

### 直接让 Codex 大规模改动

不推荐：

```text
你看着改，怎么好怎么来。
```

推荐：

```text
请先提出 2 到 3 个方案，我确认后再实现。
```

## 8. 日常开发节奏

### 每个功能分支的建议流程

```text
1. 明确需求
2. 让 Codex 阅读相关代码
3. 对齐方案
4. 小步实现
5. 本地验证
6. Review diff
7. 修复问题
8. 提交
```

### 每次提交前检查

```bash
pnpm lint
pnpm check-types
pnpm build
```

如果只是文档改动，可以至少检查：

```bash
pnpm format
```

### 每次完成后复盘

推荐问 Codex：

```text
这次改动有没有可以沉淀到 AGENTS.md 的协作规则？
有没有可以抽象成组件、工具函数或文档的内容？
有没有未验证的风险？
```

## 9. 项目演进建议

### 短期

- 将默认 Turborepo README 改成本项目专属 README
- 为 `apps/web` 建立页面级目录规范
- 明确 `packages/ui` 的组件准入标准
- 继续完善 `specs/web` 中的官网内容和页面规格

### 中期

- 增加基础测试策略
- 为关键页面增加视觉回归或截图检查
- 建立组件文档或 Storybook
- 建立 PR 模板和提交规范

### 长期

- 将业务文案、服务项目、案例数据结构化
- 为设计系统沉淀 tokens
- 建立自动化 CI：lint、typecheck、build
- 将常见开发任务整理成可复用 Codex 提示词库

## 10. 最推荐的默认工作流

你可以把下面这段作为和 Codex 协作的默认开场白：

```text
请先阅读本次任务相关代码，不要立即修改。

你需要：
1. 说明当前实现结构
2. 说明你准备改哪些文件以及原因
3. 如果存在多种方案，请列出方案让我选择
4. 遇到不确定的地方先问我
5. 实现时遵守 AGENTS.md：TypeScript、不使用 any、函数写 JSDoc、新增文件说明职责
6. 完成后运行 pnpm lint 和 pnpm check-types，并总结验证结果
```

科学的 AI 协作不是把判断权全部交给模型，而是让模型负责高强度执行、检索和整理，让人负责目标、取舍和最终判断。这样你会得到更稳定的产出，也会更快建立属于这个项目的工程秩序。
