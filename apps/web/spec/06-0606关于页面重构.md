# 需求

重构 YiForge Studio 官网的关于页面，让它从占位式理念卡片升级为工作室初心、主理人表达和 AI Native 构建理念的说明页。

# 业务背景

当前官网已经开始把「YiForge Studio 官网」作为第一个真实项目沉淀到项目页和构建记录页中。关于页也需要从项目介绍中提炼出更稳定的工作室表达。

关于页不应该重复项目页和构建记录页里的能力展示，也不应该做成传统公司介绍、简历页或服务页。它应该回答：

- 为什么会有 YiForge Studio？
- 这个工作室和主理人是什么关系？
- AI 时代对这个工作室意味着什么？
- 工作室会如何持续构建？

# 已确认判断

- 关于页最重要的是表达初心。
- 工作室代表主理人本人，主理人是 YiForge Studio 的发起者和长期构建者。
- 不单独放能力展示模块，避免和项目页、构建记录页重复。
- 不做强联系 CTA，联系入口保持克制，跳转到联系页即可。
- 关于页需要 SEO / GEO 友好，清楚表达工作室实体、主理人身份和 AI Native 定位。

# 页面定位

关于页定位为：

> YiForge Studio 的初心与自我说明页。

它不是：

- 成熟公司介绍页。
- 外包服务页。
- 个人简历页。
- 普通博客作者介绍页。

它是：

- 工作室初心页。
- 主理人说明页。
- AI Native 构建理念页。
- 长期构建方向说明页。

## 3. 这个工作室是什么

目标：解释工作室边界。

建议表达：

- 它不是外包团队。
- 它不是普通个人博客。

长期构建方向归属于这个模块内部，用简短说明和关键词 chip 表达即可，不单独做成大模块，避免页面显得像能力展示或服务介绍。

关键词：

- 真实产品
- 构建记录
- 工具与工作流
- 可上线结果

## 4. 工作室的 AI Native 开发范式

目标：解释工作室践行的 AI Native 开发理念，而不是重复能力展示，也不是只描述个人如何使用 AI。

推荐标题：

> 工作室的 AI Native 开发范式

推荐描述：

> 在 YiForge Studio 里，AI Native 开发不是把需求丢给模型生成代码，而是把问题定义、上下文、工具调用、工程实现、评测验收和持续迭代组织成一套完整的软件开发流程。

六个环节：

```txt
问题定义 -> 上下文工程 -> 流程编排 -> 协同实现 -> 评测验收 -> 上线迭代
```

这块是关于页的重要差异化模块，说明 AI 不是单点工具，而是工作室的软件开发范式。

环节说明：

- 问题定义：先确认真实场景、目标用户、成功标准和技术约束，再判断 AI 应该参与哪里。
- 上下文工程：用 spec、资料、代码、案例和问题列表组织事实依据，让模型基于上下文工作。
- 流程编排：把任务拆成模型、工具、代码执行、人工判断和安全边界都清楚的协作流程。
- 协同实现：AI 参与代码、样式、文档和调试，人持续提供反馈并控制实现方向。
- 评测验收：用类型检查、构建、页面验证和人工审阅衡量质量，而不是只看模型输出。
- 上线迭代：通过版本管理、静态部署、构建记录和反馈复盘，持续改进产品和工作流。

## 5. 工作室账本

目标：用公开、轻量的方式展示工作室第一阶段的真实投入，让“个人工作室”更具体、更可信。

推荐标题：

> 工作室账本

推荐描述：

> YiForge Studio 目前保持轻量运营，第一阶段主要投入来自域名和开发设备。

展示方式：

- 使用左右分栏或克制列表展示。
- 不写“本地开发设备”这种模糊表达，写成“本地开发设备 Mac mini M4”。
- 账本信息来自 `docs/工作室账本.md`，页面展示为静态内容。

当前条目：

- `liucodex.com 域名`：`$10 / 年`，2026
- `本地开发设备 Mac mini M4`：`¥3800`，2026

# 技术方案

新增结构化内容文件：

```txt
apps/web/content/about.ts
```

页面文件：

```txt
apps/web/app/about/page.tsx
```

样式文件：

```txt
apps/web/app/site.css
apps/web/app/browser-compat.css
```

实现要求：

- 关于页内容使用 Typed 静态数据维护。
- 页面继续静态导出，不引入数据库、CMS 或运行时内容接口。
- 保持当前导航、页脚和页面视觉系统。
- 不在关于页重复项目详情和构建记录内容，避免和项目页、构建记录页信息重复。
- 不增加新依赖。
- 移除不再使用的页面模块和样式，避免长期保留死代码。
- 首页 globe 动效只作为桌面增强体验，移动端和低动效环境不加载大体积图表资源。

# SEO / GEO 内容规则

- 页面首屏明确写出 `YiForge Studio` 是个人 AI Native 工作室。
- 页面前半部分自然回答「这是什么」「为什么成立」「如何构建」。
- 保持工作室名、域名、AI Native、主理人、官网项目等核心实体一致。
- 工作室账本中的域名、设备和年份需要保持结构化、可理解，方便搜索引擎和大模型识别。
- 使用清晰小标题和短段落，方便搜索引擎和大模型提取。
- 避免空泛口号，重要观点后面需要有具体解释。
- About 页 metadata 需要表达工作室初心、AI 协作和长期构建定位。

# 验收标准

- `/about` 不再只是占位理念卡片。
- 关于页能清楚表达工作室初心。
- 关于页能表达「工作室代表我，我是主理人」。
- 关于页能解释 AI Native 协作方式。
- 关于页不重复展示官网项目和第一篇构建记录，把项目证明留给项目页和构建记录页。
- 关于页能克制展示工作室账本，并体现轻量运营状态。
- 联系入口克制，不做强营销。
- PC 和移动端没有明显布局重叠或横向溢出。
- 首页移动端不加载 globe 图表资源，避免拖慢首屏。
- 页面没有明显无用模块、死样式或重复逻辑。
- `pnpm --filter web check-types` 通过。
- `pnpm --filter web lint` 通过。
- `pnpm --filter web build` 通过。
