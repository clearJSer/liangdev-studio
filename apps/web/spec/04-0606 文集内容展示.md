# AI补充的技术方案

## 技术结论

本次内容系统采用「Typed 项目数据 + Markdown 构建记录」方案。

官网继续保持静态导出部署，不引入数据库、CMS、服务端内容接口或运行时内容加载。项目介绍使用 TypeScript 结构化数据维护，构建记录使用 Markdown 文件维护，并在 Next.js 构建阶段生成静态列表页和详情页。

这个方案的目标是：

- 保持当前免费静态部署模式。
- 支撑后续多个项目、多篇构建记录的长期扩展。
- 让项目信息具备类型约束，减少字段缺失和维护错误。
- 让文章正文保持接近写作的体验，方便后续同步整理到微信公众号。
- 保留后续升级到 MDX、标签归档、搜索等能力的空间。

### 构建记录内容

构建记录属于长文内容，建议使用 Markdown 文件维护。

建议新增：

```txt
apps/web/content/writings/
  building-yiforge-studio-with-ai.md
apps/web/content/writings.ts
```

每篇 Markdown 文章通过 frontmatter 描述元信息：

```md
---
title: 我如何用 AI 协作搭建个人工作室官网
date: 2026-06-06
summary: 从需求文档、页面实现到内容系统，记录一次 AI 协作开发过程。
projectSlug: yiforge-studio-website
tags:
  - AI 协作
  - Next.js
  - 独立开发
---
```

正文继续使用标准 Markdown 编写。

文章字段建议包含：

- `slug`: 从文件名生成，或由 frontmatter 显式声明。
- `title`: 文章标题。
- `date`: 发布日期。
- `summary`: 列表页和 SEO 使用的摘要。
- `projectSlug`: 关联项目。
- `tags`: 标签。
- `content`: Markdown 解析后的正文内容。

### URL slug 命名规则

Markdown 文件名建议使用小写英文 kebab-case，并作为文章 URL 的 slug。

例如：

```txt
building-yiforge-studio-with-ai.md
static-markdown-content-system.md
ai-collaboration-workflow.md
```

对应 URL：

```txt
/blog/building-yiforge-studio-with-ai
/blog/static-markdown-content-system
/blog/ai-collaboration-workflow
```

文章中文标题不写在文件名里，而是写在 Markdown frontmatter 的 `title` 字段中。

例如：

```md
---
title: 我如何用 AI 协作搭建个人工作室官网
---
```

slug 不是文章的正式英文标题，只是 URL 中使用的稳定短名。命名时优先保证简短、可读、稳定，不追求完整翻译中文标题。

不建议使用随机字符串或无意义序号作为 slug，例如：

```txt
post-001.md
a1b2c3.md
article-20260606.md
```

如果文章英文命名不确定，可以先根据中文标题提炼 3 到 6 个英文关键词，或让 AI 辅助生成 slug。

## 页面方案

### 项目列表页 `/projects`

项目列表页从项目数据中读取内容，不再直接复用首页占位项目。

展示内容：

- 项目名称
- 项目摘要
- 项目状态
- 项目类型
- 技术标签
- 进入项目详情页的链接

### 项目详情页 `/projects/[slug]`

项目详情页通过 `slug` 静态生成。

页面展示：

- 项目标题和摘要
- 项目基础信息
- 技术栈
- 项目介绍正文
- 交付结果
- 关联构建记录列表

构建时需要通过 `generateStaticParams` 生成所有项目详情页，保证静态导出可用。

### 构建记录列表页 `/blog`

构建记录列表页从 Markdown 文章元信息中读取内容。

展示内容：

- 文章标题
- 发布时间
- 摘要
- 标签
- 所属项目
- 进入文章详情页的链接

### 构建记录详情页 `/blog/[slug]`

文章详情页通过 Markdown 文件静态生成。

页面展示：

- 文章标题
- 发布时间
- 标签
- 所属项目入口
- Markdown 正文
- 可选的返回项目详情页入口

构建时需要通过 `generateStaticParams` 生成所有文章详情页，保证静态导出可用。

### 关于页 `/about`

建议表达重点：

- 工作室的交付方式是「产品思考 + 前端工程 + AI 协作」。

## 静态部署约束

本需求必须遵守当前静态部署前提：

- 不使用数据库。
- 不使用 CMS。
- 不使用服务端运行时读取文章。
- 不依赖动态 API 加载内容。
- 所有项目详情页和文章详情页都在构建阶段生成。
- 新增文章后，通过重新构建和部署生成新的静态页面。

允许引入仅构建阶段使用的 Markdown 解析依赖，例如：

- `gray-matter`: 解析 Markdown frontmatter。
- `remark` / `remark-html`: 将 Markdown 转换为 HTML。

这些依赖只参与构建，不改变静态部署模式。

## 验收标准

- `/projects` 展示真实项目内容，不再只是占位项目列表。
- `/projects/yiforge-studio-website` 可以访问，并展示官网项目详情。
- `/blog` 展示真实构建记录列表。
- `/blog/building-yiforge-studio-with-ai` 可以访问，并展示 Markdown 正文。
- 项目详情页可以展示关联构建记录。
- 构建记录详情页可以返回所属项目。
- `/about` 页面从官网项目介绍中提炼工作室定位和工作方式。
- `pnpm build` 可以正常完成静态导出。
- 不引入数据库、CMS 或运行时内容服务。
- SEO metadata 能覆盖项目详情页和文章详情页。
- sitemap 后续应包含项目详情页和文章详情页。
