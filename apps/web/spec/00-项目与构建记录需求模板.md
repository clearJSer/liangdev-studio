# 内容命名规则

- 项目 slug 使用小写英文 kebab-case，例如：`yiforge-studio-website`。
- 文章 Markdown 文件名使用小写英文 kebab-case，并作为文章 URL slug，例如：`building-studio-website-with-ai.md`。
- 中文标题写在 Markdown frontmatter 的 `title` 字段中，不直接作为文件名。
- slug 优先保证简短、可读、稳定，不追求完整翻译中文标题。
- 不建议使用随机字符串或无意义序号作为 slug。

# SEO / GEO 内容规则

- 项目介绍和构建记录必须 SEO / GEO 友好。
- GEO 指 Generative Engine Optimization，也就是让内容更容易被生成式搜索、大模型问答和 AI 助手理解、引用和归纳。相关概念也可能被称为 LLMO、AEO 或 AI SEO，本文档统一使用「GEO」。
- 每个项目和每篇文章都必须有清晰、可搜索的标题。
- 每个项目和每篇文章都必须有 `summary`，用于列表页、SEO description 和社交分享摘要。
- slug 使用简短英文关键词，表达文章或项目主题。
- 页面首屏需要清楚说明当前内容是什么，避免只有抽象口号。
- 正文小标题要围绕用户可能搜索的问题组织。
- 构建记录文章需要自然包含核心关键词。
- 不堆砌关键词，不为了 SEO 牺牲真实表达。
- 项目详情页和文章详情页都需要生成独立 metadata。
- sitemap 需要包含项目详情页和文章详情页。
- 明确写出核心实体，例如工作室名称、域名、项目名称、文章主题、技术栈和作者/工作室定位。
- 在页面前半部分用自然语言回答「这是什么」「解决什么问题」「用了什么技术」「结果是什么」。
- 使用清晰的小标题和短段落，让内容可以被大模型按段落提取。
- 对关键事实保持一致表达，例如工作室名称、域名、项目 slug、技术栈、部署方式。
- 项目页要包含可被引用的事实句。
- 构建记录要包含明确的经验结论，而不只是过程叙述。
- 文章之间建立清晰关联，例如通过 `projectSlug`、相关文章列表和内部链接说明它们属于同一个项目。
- 可以使用 FAQ 或问答式小节承载高价值问题。
- 不为了 GEO 编造权威性，不写无法验证的夸大事实。

# 静态部署约束

- 不使用数据库。
- 不使用 CMS。
- 不使用服务端运行时读取文章。
- 不依赖动态 API 加载内容。
- 所有项目详情页和文章详情页都在构建阶段生成。
- 新增文章后，通过重新构建和部署生成新的静态页面。
